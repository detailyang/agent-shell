# Design: Attach/Read/Screen Render Edge Case Tests

Generated by /think on 2026-05-14
Status: Draft

## Problem

Attach 路径在实际使用中存在光标错位、颜色丢失、输出截断三类渲染问题。现有 e2e 测试对 attach 只验证 escape sequence 字节存在，不验证渲染结果的完整性；对 `read` 和 `read --screen` 路径几乎无渲染测试。需要针对三条数据路径（attach raw stream / read raw output / read --screen parsed）的边界场景补充测试，暴露根因。

## Out of Scope

- 不修 bug——本设计只负责暴露问题，修复是后续工作
- 不覆盖 attach picker TUI、Ctrl-C/Ctrl-D 退出等交互逻辑
- 不覆盖 VteGrid 内部实现测试（那是 unit test 范畴）
- 不覆盖 recording/replay 渲染正确性

## Constraints

- 测试必须跑在 e2e 框架内（`crates/e2e/tests/integration.rs`），使用 `DaemonHandle`
- 每个测试独立 daemon 实例（`start_daemon()`），不跨测试共享状态
- 并行安全：4 线程并行，每个测试有独立的 `AGENT_SHELL_HOME`
- 单测试超时 ≤15s（硬限制：`cargo test` 默认 60s，留余量）
- 测试环境有 bash，不假设 zsh/fish

## Assumptions

- `read --screen` 的 VteGrid 设计上丢弃 SGR 属性——颜色测试在 screen 路径只验证文本位置正确性，不验证颜色
- `read` 不带 `--client-id` 返回全量 ringbuf，带 `--client-id` 返回增量——两种模式都需测试
- attach handshake phase（base64 初始输出）和 attach live stream phase 是同一个 ringbuf 的两种读取方式——都可能出截断

## Approach

按数据路径分三个测试模块，每个模块覆盖颜色完整性、光标正确性、截断/边界。

新增一个共享 helper `assert_render`，根据路径类型自动选择验证策略：

```
enum RenderPath {
    Attach,        // 验证 raw bytes 中 escape sequence 完整性
    ReadRaw,       // 验证 read 返回的 output 字段中 escape sequence 完整性
    ReadScreen,    // 验证 read --screen 返回的 screen 行文本 + cursor 位置
}
```

数据流：

```
                  ┌─ attach (raw bytes via base64 handshake + live stream)
  PTY → ringbuf ──┼─ read (raw bytes via JSON output field)
                  └─ read --screen (VteGrid parsed → screen[] + cursor)
```

### 模块 1: `attach_render`（5 个测试）

覆盖 attach raw stream 的颜色/光标/截断边界。

| 测试 | 场景 | 验证 |
|------|------|------|
| `attach_color_after_prompt` | 先 send 产生 prompt 输出，再 attach，验证 base64 handshake 输出中 SGR 序列完整 | initial_output 含 `\x1b[31m` + `RED` + `\x1b[0m` |
| `attach_cursor_after_command` | 先 send 产生光标移动命令（printf 定位），attach 验证 initial_output 中 cursor 定位序列完整 | initial_output 含 `\x1b[row;colH` |
| `attach_truncated_escape` | 先 send 产生大量输出（>4KB），再用小 buffer-size session attach，验证 base64 解码后不会在 escape sequence 中间截断 | 解码后的字节不以不完整的 `\x1b` 结尾（即 `\x1b` 后必须跟 `[` 或 `]`） |
| `attach_live_color_during_command` | attach 后通过 attach 写入产生颜色的命令，验证 live stream 中 SGR 完整 | stream output 含 `\x1b[32m` + text |
| `attach_resize_renders_correctly` | attach 中 resize，通过 attach 发送带光标定位的输出，验证 escape sequence 在新尺寸下仍完整 | stream output 含 resize 后的 cursor 定位序列 |

### 模块 2: `read_render`（5 个测试）

覆盖 `read` 命令返回的 raw output 的颜色/光标/截断边界。

| 测试 | 场景 | 验证 |
|------|------|------|
| `read_color_preserved` | send 产生 SGR 颜色输出，read 验证 output 字段含完整 escape sequence | output 含 `\x1b[31m` + text + `\x1b[0m` |
| `read_cursor_position_preserved` | send 产生光标定位输出，read 验证 output 含 `\x1b[row;colH` | output 含 escape sequence |
| `read_incremental_color` | 用 `--client-id` 分两次 read，第一次取前半段，第二次取含颜色的后半段 | 第二次 read 的 output 含 SGR 序列，不被截断 |
| `read_small_buffer_truncation` | 小 buffer-size session 产生大量输出后 read，验证 output 不在 escape sequence 中间截断 | output 末尾不以孤立的 `\x1b` 结尾 |
| `read_after_resize` | resize 后 send 带光标定位的输出，read 验证 escape sequence 完整 | output 含 resize 后的 cursor 定位 |

### 模块 3: `screen_render`（5 个测试）

覆盖 `read --screen` 返回的 VteGrid 解析结果的正确性。

| 测试 | 场景 | 验证 |
|------|------|------|
| `screen_text_after_cursor_move` | send 产生光标移动+写入，`read --screen` 验证文本落在正确行列 | `screen[row]` 含写入的文本，cursor 位置正确 |
| `screen_multiline_colored_text` | send 产生多行带颜色的文本，`read --screen` 验证文本完整（颜色被丢弃是预期的） | screen 行含完整文本，不含 escape sequence 残留 |
| `screen_clear_then_write` | send 产生 clear screen + 写入，`read --screen` 验证清屏后只有新内容 | 旧行被清除，只有新写入的文本 |
| `screen_after_resize` | resize 后 send 多行输出，`read --screen` 验证 screen 行数与新 rows 一致 | `screen.len() == new_rows`，文本在正确位置 |
| `screen_cursor_after_erase` | send 产生 erase line + 写入，`read --screen` 验证被擦除区域为空、新写入正确 | 擦除后的列为空格，新文本在正确位置 |

## Key Decisions

1. **按数据路径分模块而非按问题域分模块**——颜色/光标/截断的根因在不同路径上不同（attach 是 base64 边界，read 是 client cursor 边界，screen 是 VteGrid 解析），按路径分让失败定位更快。这成立的前提是每条路径的边界场景确实不同，目前看是成立的。

2. **"不在 escape sequence 中间截断"用启发式验证**——检查末尾不以孤立 `\x1b` 结尾（`\x1b` 后必须跟 `[` 或 `]`）。这是最小代价的截断检测。更严格的验证需要完整 VTE 解析器，对测试来说过度。只要启发式能暴露截断 bug 就够了。

3. **screen 路径不验证颜色**——VteGrid 设计上丢弃 SGR 属性，验证颜色是 attach/read 路径的职责。screen 路径只验证文本位置和完整性。如果未来 VteGrid 保留 SGR，需增加 screen 颜色测试。

4. **不做录制回放测试**——recording/replay 路径的渲染问题与这三条路径正交，且录制 fixture 依赖环境，不可移植。单独设计。

5. **新 helper `assert_render` 放在 `e2e/src/lib.rs`**——跨测试模块共享，避免重复。复用现有的 `attach_after_send` 模式。

## Failure Modes & Mitigations

| 失败模式 | 缓解 |
|---------|------|
| 测试不稳定：颜色 escape sequence 在不同 bash 版本下行为不同 | 只用 `printf` 产生 escape sequence，不依赖 `echo -e` 或 `ls --color` |
| 截断启发式误报：合法输出恰好在 `\x1b` 处结束 | 检查 `\x1b` 是否在末尾 1 字节且后面无 `[` 或 `]`。正常输出几乎不可能以裸 `\x1b` 结尾 |
| `read --screen` 测试 flaky：VteGrid 状态依赖 feed 时机 | 每次 `read --screen` 前先 `send` 一个轻量命令（`echo marker`）确保 feed 已执行 |
| 并行测试 daemon 冲突 | 已有 `AGENT_SHELL_HOME` 隔离，每个 `start_daemon()` 独立 temp dir |

## Test Coverage

**Happy path:**
- attach: handshake 含颜色、attach live stream 含颜色 — `attach_color_after_prompt`, `attach_live_color_during_command`
- read: 全量读取含颜色 — `read_color_preserved`
- screen: 光标移动后文本位置 — `screen_text_after_cursor_move`

**Error/boundary:**
- ringbuf 小 buffer 截断 — `attach_truncated_escape`, `read_small_buffer_truncation`
- 增量 read 截断 — `read_incremental_color`
- resize 后渲染 — `attach_resize_renders_correctly`, `read_after_resize`, `screen_after_resize`
- clear/erase 后状态 — `screen_clear_then_write`, `screen_cursor_after_erase`

**Gap（本设计不覆盖）:**
- attach 连接断开后重连的渲染一致性
- 多 attach 并发写入的渲染
- 非 UTF-8 字节（纯二进制）的传输完整性

## Rollback Plan

纯测试代码，删除新增的三个模块及 helper 即可。无生产代码变更，零风险。

## Open Questions

无。所有设计决策已在 Key Decisions 中明确。

## Success Criteria

1. 新增 15 个 e2e 测试，覆盖 3 条路径 × 5 个场景
2. 所有新测试在当前代码上通过（颜色/光标场景）或暴露已知 bug（截断场景可能 fail，这是预期）
3. 截断场景如果 fail，fail message 必须包含足够信息定位截断位置（输出末尾 hex dump）
4. 全套 e2e 测试（新+旧）在 4 线程并行下 ≤45s 完成

## Implementation Steps

1. 在 `crates/e2e/src/lib.rs` 添加 `RenderPath` enum 和 `assert_render` 函数。`assert_render` 签名：`fn assert_render(daemon: &mut DaemonHandle, sid: &str, printf_cmd: &str, path: RenderPath, expected_escape: &[u8], expected_text: &str)`。实现：根据 path 调用 `connect_attach_rw` / `cli_json("read")` / `cli_json("read --screen")`，获取输出后断言 escape sequence 和文本。

2. 在 `crates/e2e/src/lib.rs` 添加 `assert_no_truncated_escape(bytes: &[u8])` 函数。逻辑：检查 bytes 末尾 2 字节，如果倒数第 1 字节是 `\x1b` 且后面无 `[` 或 `]`，则 fail 并打印末尾 20 字节 hex dump。

3. 在 `crates/e2e/tests/integration.rs` 添加 `mod attach_render` 模块，实现 5 个测试。每个测试调用 `start_daemon()` + `create --shell /bin/bash`，用 `assert_render` 或直接操作 `AttachConnection`。`attach_truncated_escape` 需 `--buffer-size 4096`。

4. 在 `crates/e2e/tests/integration.rs` 添加 `mod read_render` 模块，实现 5 个测试。每个测试用 `daemon.cli_json(&["read", ...])` 读取输出并断言。`read_small_buffer_truncation` 需 `--buffer-size 4096`。

5. 在 `crates/e2e/tests/integration.rs` 添加 `mod screen_render` 模块，实现 5 个测试。每个测试用 `daemon.cli_json(&["read", "--screen", ...])` 读取 screen 和 cursor，断言行文本和光标位置。

6. 运行 `cargo test --release -p agent-shell-e2e`，确认 151 个测试全部通过（136 旧 + 15 新）。如果截断测试失败，确认 fail message 包含 hex dump，记录为已知 bug。

7. `cargo build --release`，复制到 `~/.local/bin/`。
