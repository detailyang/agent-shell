# Design: Replay TUI Fix

Generated by /think on 2026-05-19
Status: Draft

## Problem

`agent-shell replay` 将录制的原始 PTY 字节流直接写入 stdout，没有进入 raw mode、没有过滤终端查询序列、没有还原原始终端尺寸。当录制内容包含 vim 等 TUI 应用时，出现三层叠加的渲染错误：

1. `\x1b[6n`（DSR）被写入当前终端，终端向 stdin 发送 CPR 响应，响应无人消费，在非 raw mode 下被 echo 为可见乱码。
2. `\x1bPzz\x1b\\`（DCS）同理触发终端响应。
3. `\x1b[1;24r` 将滚动区域锁定为原始会话的行数（24 行），若当前终端行数不同，vim 的所有绝对定位命令（`\x1b[24;1H` 等）全部错位。

根本缺陷是录制文件没有保存原始会话的 rows/cols，导致 replay 无法知道目标终端尺寸，也无法做任何尺寸适配。

## Out of Scope

- `agent-shell replay` 的 CLI 接口（参数、输出格式）不变。
- `--dump` 模式不变（用于非 TTY 环境的原始字节输出）。
- 已有录制文件的格式迁移（旧文件无 header，降级为启发式处理）。
- mouse event 录制回放的正确性（超出本次范围）。
- Windows 平台支持。

## Constraints

- CLI 接口冻结：不新增参数，不改变退出码语义。
- 录制文件格式变更必须向后兼容：旧文件（无 header）可继续 replay，降级为启发式尺寸推断。
- 实现限于 `crates/core/src/recording.rs` 和 `crates/cli/src/main.rs`，不改 daemon 侧逻辑以外的部分（daemon 侧仅需在 `Recording::new` 后写一行 header）。

## Assumptions

1. 录制文件第一行若存在 header，其 `dir` 字段值为 `"meta"`，replay 端以此区分新旧格式。若 `dir != "meta"`，视为旧文件。
2. 当前终端支持 `TIOCSWINSZ` ioctl（macOS/Linux 均支持）。
3. replay 只在 TTY 环境下运行时才进 raw mode 和 resize；非 TTY 环境（pipe/CI）保持现有行为不变。
4. DSR（`\x1b[6n`）和 DCS（`\x1bP...\x1b\\`）过滤不影响 TUI 应用的视觉输出，因为这些序列是终端能力探测，不产生可见字符。
5. 启发式尺寸推断：从输出字节流中找第一个 `\x1b[{top};{bottom}r` scroll region 序列，用 `bottom` 作为行数；cols 从 `\x1b[{row};{col}H` 中取最大 col 值。无法推断时使用当前终端尺寸（不 resize）。

## Approach

### 录制侧：写入 metadata header（daemon，`session.rs`）

`Session::new` 创建 `Recording` 后，立即调用新方法 `Recording::write_header(rows, cols, program)`，写入第一行：

```json
{"dir":"meta","ts":1234567890000,"rows":24,"cols":80,"program":"vim"}
```

`RecordingEvent` 结构新增可选字段，或新增独立 `RecordingHeader` 结构体（推荐后者，避免污染事件类型）。

### replay 侧：读取 header + raw mode + DSR/DCS 过滤 + resize（`recording.rs`，`main.rs`）

`timed_replay` 执行流程变更：

```
timed_replay(reader, speed)
  │
  ├─ 1. read_header(events[0])
  │     ├─ if dir == "meta" → ReplayMeta { rows, cols, program }
  │     └─ else             → None（旧文件，启发式推断）
  │
  ├─ 2. if stdout is TTY:
  │     ├─ enter_raw_mode()         → RawModeGuard（RAII 还原）
  │     ├─ save_terminal_size()     → (orig_rows, orig_cols)
  │     └─ resize_terminal(target_rows, target_cols)  → TIOCSWINSZ
  │
  ├─ 3. 逐 group 写出，写出前过滤 DSR/DCS：
  │     filter_query_sequences(buf) → 移除 \x1b[6n 和 \x1bP...\x1b\\
  │
  ├─ 4. replay 结束（正常或 Ctrl-C）：
  │     ├─ RawModeGuard::drop()     → 还原 termios
  │     └─ restore_terminal_size(orig_rows, orig_cols)  → TIOCSWINSZ
  │
  └─ 5. 若 alternate screen 未退出（录制以 crash 结尾）：
        写出 \x1b[?1049l 强制退出 alternate screen
```

### 尺寸决策逻辑（`resolve_replay_size`）

```
if header exists:
    target = (header.rows, header.cols)
else:
    target = heuristic_size(all_out_bytes)
    if heuristic fails:
        target = current_terminal_size()
```

### 组件边界

```
crates/core/src/recording.rs
  ├─ RecordingHeader { rows, cols, program }       ← 新增
  ├─ Recording::write_header(rows, cols, program)  ← 新增
  ├─ read_header(first_line) → Option<RecordingHeader>  ← 新增
  ├─ filter_query_sequences(buf: &[u8]) -> Vec<u8>      ← 新增
  ├─ heuristic_size(bytes: &[u8]) -> Option<(u16,u16)>  ← 新增
  └─ timed_replay()                                ← 修改

crates/core/src/session.rs
  └─ Session::new()  ← 修改：在创建 Recording 后调用 write_header

crates/cli/src/main.rs
  └─ timed_replay 调用路径中注入 raw mode + resize  ← 修改
     （或将 raw mode 逻辑下沉到 recording.rs，保持 main.rs 不变）
```

raw mode 和 terminal resize 逻辑下沉到 `recording.rs`（`timed_replay` 内部处理），`main.rs` 调用接口不变。`enter_raw_mode` 和 `RawModeGuard` 从 `main.rs` 提取到 `core` crate 的独立模块 `crates/core/src/terminal.rs`，供 `recording.rs` 和 `main.rs` 共用。

## Key Decisions

1. **选 `RecordingHeader` 独立结构体而非扩展 `RecordingEvent`**。`RecordingEvent` 的 `data` 字段是 base64，混入数值字段会破坏类型一致性，增加 serde 复杂度。独立结构体通过 `dir == "meta"` 识别，旧代码遇到未知 `dir` 直接跳过，天然向后兼容。此判断在旧格式文件消失之前一直有效。

2. **DSR/DCS 过滤在 replay 侧做，不在录制侧做**。录制侧需要保留完整字节流以备其他用途（审计、diff）。过滤是 replay 的渲染策略，不是录制的语义。此决策在"录制文件用于审计"的需求存在时永久成立。

3. **raw mode 和 resize 逻辑下沉到 `recording.rs`，而非留在 `main.rs`**。`main.rs` 的 `timed_replay` 调用接口不变，replay 的终端处理是其内部实现细节。此决策在 replay 不需要从 CLI 层接收额外终端参数时成立。

4. **replay 结束时无条件发 `\x1b[?1049l`**。检测"是否进入了 alternate screen"需要解析整个字节流，代价高且容易出错。无条件发送幂等序列（若未进入 alternate screen，该序列无副作用）更可靠。此决策在终端对幂等 `\x1b[?1049l` 行为符合 ECMA-48 标准时成立（macOS Terminal、iTerm2、kitty 均符合）。

5. **`RawModeGuard` 提取到 `core` crate 而非在 `main.rs` 和 `recording.rs` 各自实现**。两处均需 raw mode，重复实现是维护负担。提取到 `crates/core/src/terminal.rs` 后，两处共用同一 RAII guard。此决策在未来没有第三处需要 raw mode 时成立（实际上越多越合算）。

## Failure Modes & Mitigations

| Failure Mode | Mitigation |
|---|---|
| replay 终端尺寸与原始不同 | 从 header 读取精确尺寸；旧文件降级为启发式推断；推断失败时使用当前终端尺寸（不 resize，而非崩溃） |
| DSR `\x1b[6n` 触发当前终端响应 | `filter_query_sequences` 在写出前移除所有 `\x1b[6n`；CPR 响应不会出现在 stdout |
| DCS/OSC 触发终端响应 | `filter_query_sequences` 同时移除 `\x1bP...\x1b\\`（DCS）和 `\x1b]...\x07/\x1b\\`（OSC title set） |
| replay 结束后终端状态损坏（alternate screen 未退出） | replay 结束时无条件写出 `\x1b[?1049l`；`RawModeGuard` RAII 还原 termios；`restore_terminal_size` 还原 TIOCSWINSZ |
| 非 TTY 环境（pipe/CI） | `if stdout is TTY` 判断：非 TTY 时跳过 raw mode、resize、DSR 过滤，保持现有行为 |

## Rollback Plan

- `recording.rs` 和 `session.rs` 的改动均向后兼容。回滚只需 revert 相关 commits。
- 已写入 header 的录制文件无法回滚为旧格式，但旧版 replay 代码遇到 `dir == "meta"` 的第一行时，会因为无法 base64 decode `data` 字段（header 无此字段）而跳过该行，不会崩溃——旧代码对新文件的降级行为：丢失第一帧（meta 行），其余正常播放。
- 若需完全隔离，可加 feature flag `replay-tui-fix`，默认关闭，验证后再开启。

## Implementation Steps

1. **新增 `crates/core/src/terminal.rs`**  
   将 `RawModeGuard` 和 `enter_raw_mode` 从 `crates/cli/src/main.rs` 提取到此文件。新增 `save_terminal_size() -> Option<(u16, u16)>` 和 `set_terminal_size(rows: u16, cols: u16) -> bool`（封装 `TIOCSWINSZ` ioctl）。在 `crates/core/src/lib.rs` 中 `pub mod terminal;`。  
   验证：`crates/cli/src/main.rs` 改为 `use agent_shell_core::terminal::enter_raw_mode`，`cargo build` 通过。

2. **新增 `RecordingHeader` 结构体和 `Recording::write_header`（`recording.rs`）**  
   ```rust
   pub struct RecordingHeader { pub rows: u16, pub cols: u16, pub program: String }
   ```
   `write_header` 序列化为 `{"dir":"meta","ts":<ms>,"rows":<r>,"cols":<c>,"program":<p>}` 写入文件第一行（在任何 event 之前）。  
   验证：单元测试 `header_is_first_line`：创建 Recording，调用 write_header，读取文件第一行，断言 `dir == "meta"` 且字段正确。

3. **`Session::new` 调用 `write_header`（`session.rs`）**  
   在 `Recording::new(rec_path).ok()` 成功后，立即调用 `rec.write_header(rows, cols, &program_display)`。  
   验证：e2e 测试 `create_with_record_has_header`：创建带 `--record` 的 session，读取录制文件第一行，断言 header 字段与 session 参数一致（AC-009）。

4. **新增 `read_header`、`filter_query_sequences`、`heuristic_size`（`recording.rs`）**  
   - `read_header(line: &str) -> Option<RecordingHeader>`：若 `dir == "meta"` 解析并返回，否则 `None`。  
   - `filter_query_sequences(buf: &[u8]) -> Vec<u8>`：移除 `\x1b[6n`（DSR）和 `\x1bP...\x1b\\`（DCS，贪婪匹配到 String Terminator）。不过滤 OSC title（`\x1b]0;...\x07`），因为 OSC title 不触发 stdin 响应。  
   - `heuristic_size(bytes: &[u8]) -> Option<(u16, u16)>`：扫描第一个 `\x1b[{top};{bottom}r`，返回 `(bottom, max_col)`，其中 `max_col` 取所有 `\x1b[{r};{c}H` 中最大的 `c` 值，若无则返回 `None`（cols 部分）。  
   验证：单元测试分别覆盖 filter（含 DSR 输入 → 输出无 DSR）、heuristic（含 scroll region 输入 → 正确推断行数）。

5. **修改 `timed_replay`（`recording.rs`）**  
   函数签名不变。执行流程按 Approach 节描述修改：  
   - 读取并分离 header（若第一行 `dir == "meta"` 则消费，不加入 event 列表）。  
   - `resolve_replay_size`：优先 header，其次 heuristic，其次当前终端尺寸。  
   - `if unsafe { libc::isatty(1) != 0 }`：进 raw mode，保存当前尺寸，TIOCSWINSZ resize。  
   - 每个 group 写出前调用 `filter_query_sequences`（仅 TTY 模式）。  
   - replay 结束（loop 结束或 `?` 传播的 IO error 前）：写出 `\x1b[?1049l`，还原终端尺寸，`RawModeGuard` drop 还原 termios。  
   验证：`cargo test` 全量通过（含已有 `timed_replay_merges_split_escape_sequence` 回归测试）。

6. **手工验证 AC-001、AC-002、AC-007、AC-008**  
   - AC-002：`agent-shell replay ~/.agent-shell/recordings/bfd784c5.jsonl`，目视确认 vim 渲染正确，退出后终端正常。  
   - AC-007：构造一个截断的录制文件（只有 `\x1b[?1049h`，无 `\x1b[?1049l`），replay 后确认 `echo ok` 正常输出。  
   - AC-008：`agent-shell replay <file> | xxd | grep -c '1b 5b 36 6e'` 应大于 0（非 TTY 时不过滤）。
